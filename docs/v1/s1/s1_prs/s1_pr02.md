# PR-02: Turn Budget Guardrails + Explicit Bounded-Failure Semantics

## Goal

Enforce configuration-driven turn budgets so limit exhaustion ends turns deterministically with explicit, auditable bounded-failure contracts.

## Builds On

PR-01 merged.

## Acceptance

- Slice 1 turn-budget thresholds are configuration-driven with defaults: `max_recent_turns=12`, `max_context_tokens=6000`, `max_response_tokens=700`, `max_model_attempts=2`, `max_turn_wall_time_ms=20000`.
- when any configured budget is exhausted (context, response, model-attempt, or wall-time), the turn terminates in failed status and returns bounded-failure semantics with error code `E_TURN_LIMIT_REACHED`; for the current HTTP surface this maps to a consistent 4xx response.
- bounded-failure responses include structured, auditable limit details that identify which budget was exhausted and the measured/applied values relevant to that failure.
- user-facing bounded-failure messaging is clear about why the turn stopped and what is still needed to proceed; exact copy is non-normative as long as the machine-readable contract remains stable.
- unresolved interaction loops are bounded by the same global turn limits and terminate with `E_TURN_LIMIT_REACHED`; no dedicated clarification counter is introduced.
- event chains preserve observability of decision outcome and terminal result for both successful and bounded-failure turns, including explicit user-visible bounded-failure emission before terminal `evt.turn.failed`, with no dangling `in_progress` turn state.
- bounded-limit failures remain distinct from provider/network/model errors and are emitted only when a configured turn budget is actually exhausted.

## Non-Goals

- does not change provider portability or failover strategy.
- does not introduce capability execution, approvals, or new capability domains.
- does not implement session rotation or cross-session durable memory behavior.
- does not add brittle response-type classifiers or clarification-specific counters.
- does not redefine existing Slice 1 context ordering semantics beyond applying budget guardrails around turn execution.

## Key Decisions (if any beyond L2)

**Transport-agnostic bounded-failure contract**: `E_TURN_LIMIT_REACHED` semantics are normative across surfaces; HTTP status mapping is an API-layer mapping choice and not the product-level contract.

**Dual-channel bounded-failure observability**: bounded failures must be both user-visible (assistant/event emission) and machine-auditable (terminal failed state + structured limit metadata), so replay and audit can reconstruct exactly what happened.

**Current HTTP + event mapping**: for the current API surface, bounded turn-limit failures map to HTTP `429` and emit `evt.turn.limit_reached` before `evt.assistant.emitted` and terminal `evt.turn.failed`.
