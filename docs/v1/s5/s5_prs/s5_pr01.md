# PR-01: Canonical Memory Core + Explicit Session Rotation + Cross-Session Recall

## Goal

Deliver the first end-to-end Slice 5 continuity path with canonical durable memory, read-only memory projection, explicit session rotation, and validated cross-session recall.

## Builds On

Slice 4 PR-03 merged state (bounded turn orchestration, surfaced response-contract enforcement, and policy/audit runtime already in place).

## Acceptance

- Ariel persists durable memory in Postgres as canonical state with typed memory classes and required provenance/verification metadata.
- Ariel exposes a dedicated read-only memory projection surface that is derived from canonical state and remains redaction-safe and contract-enforced.
- Explicit user-provided facts/preferences/commitments can be captured as validated durable memory through normal conversation turns with auditable outcomes.
- Memory mutation remains conversation-mediated in this PR; no generic memory CRUD write endpoint is introduced.
- User can explicitly start a new conversation through a dedicated rotation-intent surface; Ariel rotates sessions safely, preserves exactly one active session, marks the prior session inactive, and records `user_initiated` rotation reason.
- Existing session bootstrap/status behavior remains backward-compatible and non-rotating by default unless explicit rotate intent is provided.
- Rotation intent handling is idempotent and race-safe, so duplicate rotate submissions do not create multiple active sessions or duplicate user-visible rotation outcomes.
- Session rotation writes bounded continuity artifacts (including episodic continuity and open-commitment context) so future turns do not require full transcript replay.
- New-session turns can recall relevant validated memory (including open commitments) from prior sessions, while candidate-only memory is excluded from recall.
- Turn context remains deterministic and bounded after memory integration; recall uses explicit top-k bounded selection with stable ordering and does not bypass existing turn-budget behavior.
- Regression coverage demonstrates continuity across explicit rotation plus restart, projection/canonical consistency for read paths, and no regressions to existing surfaced session/timeline contracts.

## Non-Goals

- does not add deterministic automatic threshold-triggered session rotation.
- does not add inferred `candidate` memory promotion workflows.
- does not add correction/removal lifecycle behavior beyond baseline validated capture.
- does not add generic memory CRUD write APIs, autonomous memory rewriting, or Nexus sync-authority changes.

## Key Decisions (if any beyond L2)

**Explicit rotate-intent contract**: Rotation is initiated only through dedicated rotate intent (or future threshold policy), while existing session bootstrap/status flows remain non-rotating by default.

**Idempotent rotation semantics**: Duplicate rotate requests must collapse safely to one durable rotation outcome and preserve the one-active-session invariant under concurrency.

**Fail-closed surfaced contract expansion**: Any new memory projection and rotation observability added in this PR must be enforced through strict surfaced response contracts rather than permissive payload passthrough.
