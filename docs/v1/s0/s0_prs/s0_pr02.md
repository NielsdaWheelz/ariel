# PR-02: Private Tailnet Access + Restart Durability Hardening

## Goal

Make Slice 0 usable day-to-day from phone over private tailnet routing only, while proving restart-safe conversation continuity and auditable model-failure behavior.

## Builds On

PR-01 merged.

## Acceptance

- Ariel is reachable from phone via HTTPS on the node's private tailnet DNS name, and the app service itself is localhost-only.
- Deployment state shows no public ingress path for Ariel (no public funnel/exposure), and the private-only posture is verifiable.
- Tailnet policy allows access only for explicitly approved identities/devices; non-allowed identities cannot reach the Ariel surface.
- Runtime model calls use configured real credentials from server-side secret wiring; missing/invalid credentials fail clearly without leaking secret values.
- After service restart, prior active-session conversation history and timeline remain visible, and new turns append to that same active session.
- Model-provider failure paths still emit auditable model/timeline events and terminate the turn with explicit failed state and reason.
- Deployment and run workflow documents a repeatable private self-hosted setup, including verification steps for ingress posture and restart continuity.

## Non-Goals

- No capability execution, approval workflow, or async job implementation.
- No cross-session durable memory retrieval or session rotation policy changes.
- No multi-user tenancy or public internet hosting support.
- No provider portability/failover hardening beyond making one configured provider deployment-safe.
- No chat/timeline UX redesign beyond preserving existing Slice 0 behavior in private deployment.

## Key Decisions (if any beyond L2)

**Private ingress contract is enforced at the network boundary**: Ariel remains localhost-bound and is exposed only through tailnet HTTPS mediation, not direct public listeners.

**Fail-closed access posture**: access defaults to deny and is granted only to explicitly allowlisted tailnet identities.

**Credential handling remains server-only**: provider secrets are resolved at runtime on the host and never embedded in client code, repo-tracked values, or timeline payloads.

**Restart durability is a release gate**: continuity of history and active-session append semantics across restarts is mandatory in PR-02, not deferred.
