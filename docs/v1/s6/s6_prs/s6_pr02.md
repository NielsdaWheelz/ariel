# PR-02: Maps Read Vertical + Service-Credential Reliability

## Goal

Deliver production-safe Maps directions and nearby-place workflows as allowlisted `read` capabilities with deterministic clarification behavior, citation-ready retrieval outputs, and robust typed failure handling.

## Builds On

PR-01 merged state.

## Acceptance

- Ariel introduces `cap.maps.search_places` and `cap.maps.directions` as allowlisted `read` capabilities with no approval path.
- Maps capability contracts are narrow and explicit (versioned capability identity, strict input/output validation, and retrieval-style output boundaries).
- Maps execution uses restricted server-side provider credentials only (no user reconnect/consent loop), with encrypted-at-rest secret handling and rotation-safe runtime semantics.
- Missing/invalid Maps credential or configuration state returns stable typed recoverable failures (`provider_credentials_missing`, `provider_credentials_invalid`) with explicit operator/user recovery guidance.
- Provider/runtime failures map to stable typed outcomes (`provider_timeout`, `provider_network_failure`, `provider_rate_limited`, `provider_upstream_failure`, `provider_permission_denied`, `provider_request_rejected`, `provider_invalid_payload`, `provider_unreachable`) with actionable recovery guidance and auditable lifecycle reasons.
- Directions responses are grounded in provider evidence, include user-visible provenance/citation artifacts, and keep auditable action lifecycle visibility.
- Nearby-place responses return useful candidate places with disambiguating metadata, user-visible provenance/citation artifacts, and explicit uncertainty when evidence is insufficient.
- `cap.maps.directions` asks explicit clarification when required route fields (origin/destination) are missing, and does not infer location from implicit IP/device geolocation.
- `cap.maps.search_places` asks explicit clarification when required place-query location context is missing, and does not infer location from implicit IP/device geolocation.
- Maps retrieval outputs remain normalized and citation-friendly so grounded synthesis works consistently in single- and mixed-retrieval turns.
- Egress intent preflight remains fail-closed with explicit destination allowlists; undeclared or non-allowlisted egress is blocked before execution.
- Maps capability execution remains isolated from Google OAuth connector readiness/consent state used by calendar/email/drive.
- Policy and runtime invariants remain intact (allowlisted `read`, no approval path, integrity checks, redaction boundaries, auditable lifecycle events).
- Regression coverage validates mixed retrieval turns (Drive/Maps/web/news), typed Maps failure contracts, deterministic clarification behavior, and policy invariants.

## Non-Goals

- does not add traffic-aware route optimization, multi-stop planning, or commute learning/personalization.
- does not add proactive maps-triggered notifications.
- does not add write-side Maps actions or approval-gated Maps side effects.
- does not change Drive capability behavior, OAuth reconnect semantics, or approval policy boundaries.
