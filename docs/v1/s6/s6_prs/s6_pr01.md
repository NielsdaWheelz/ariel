# PR-01: Drive Vertical (Search/Read/Share) + OAuth Scope Expansion

## Goal

Deliver a production-safe Drive vertical slice so Ariel can find files, inspect bounded file content, and perform approval-gated sharing under existing Google policy, approval, audit, and typed-failure contracts.

## Builds On

Slice 4 PR-03 merged state (Google connector/readiness foundations) and Slice 5 PR-01 merged state (current orchestration and surfaced response contracts).

## Acceptance

- Ariel introduces `cap.drive.search` and `cap.drive.read` as allowlisted `read` capabilities, and `cap.drive.share` as `external_send` with approval required.
- Drive capability contracts remain narrow and explicit (versioned capability identity, strict input/output validation, and bounded read outputs).
- Drive reconnect intent is capability-scoped and least-privilege (incremental scope upgrade for Drive operations) while preserving current connector status/readiness semantics.
- Drive search returns relevant file candidates for natural-language queries with enough user-visible metadata to disambiguate results and inspect lifecycle outcomes.
- Drive read returns a bounded, user-meaningful content view with user-visible provenance/citation artifacts; unsupported, too-large, and unavailable paths are typed outcomes with explicit recovery guidance.
- Drive read/search outputs remain normalized retrieval-style results compatible with grounded answer synthesis, rather than raw provider payload passthrough.
- Drive failure behavior is deterministic and typed across auth/scope plus provider/transient/quota/permission classes, always with explicit recovery guidance and no silent unsafe fallback.
- Drive share does not execute before approval and, after approval, executes only once for the exact approved payload hash; deny/expire/mismatch paths remain blocked and auditable.
- Egress allowlists, execution-integrity checks, redaction, and surfaced action lifecycle visibility remain intact for all new Drive capabilities.

## Non-Goals

- does not add Maps capabilities (`cap.maps.*`) in this PR.
- does not add Drive upload/create/edit/delete/move workflows.
- does not promise full-fidelity rendering across every binary or Google-native document format.
- does not introduce approval bypass, autonomous sharing, or any external-send auto-execution.

## Key Decisions (if any beyond L2)

**Failure-contract-first**: Drive launches only with stable typed failure contracts and recoverability guidance across auth/scope and provider/runtime failure paths.

**Retrieval contract continuity**: Drive read/search must produce citation-friendly retrieval outputs that preserve existing grounded-answer and provenance behavior across mixed retrieval turns.

**Share safety boundary**: `cap.drive.share` remains a hard `external_send` boundary with exact-payload approval semantics, independent from read/search capability success.
