# PR-03: Connector Readiness Semantics + Attendee Consent Closure

## Goal

Close Slice 4 readiness and attendee-consent gaps so reconnect guidance is deterministic and connector status matches runtime auth outcomes.

## Builds On

PR-02 merged.

## Acceptance

- when any calendar/email capability execution fails with `consent_required`, `scope_missing`, or `access_revoked`, connector readiness surfaces as `reconnect_required` and remains so until a successful reconnect flow completes.
- transient retryable failures (for example temporary upstream timeout/network/provider instability) do not remap a healthy connected connector to `reconnect_required`.
- readiness remapping uses an explicit blocking-vs-transient classification contract so auth/scope outcomes cannot be interpreted inconsistently across capabilities or turns.
- reconnect supports deterministic attendee-consent remediation for slot planning: attendee-aware slot-planning intent requests attendee free/busy consent while preserving already granted scopes and avoiding unrelated scope escalation.
- once attendee free/busy consent is granted, attendee-inclusive slot proposals use attendee intersection behavior and stop presenting fallback-only guidance.
- when attendee free/busy consent is not granted, attendee slot planning remains explicit user-calendar-only fallback with clear reconnect guidance.
- connector status/readiness transitions and reconnect scope requests remain auditable, and acceptance regression coverage proves transition behavior plus attendee-consent closure paths.

## Non-Goals

- does not add new Google capability IDs beyond the existing Slice 4 set.
- does not change approval policy boundaries for calendar create, email draft, or email send.
- does not add Drive/Maps workflows or proactive Google-triggered notification behavior.
- does not introduce autonomous/background reconnect loops or approval bypass behavior.

## Key Decisions (if any beyond L2)

**Readiness persistence contract**: `reconnect_required` is treated as user-action-required state for blocking auth/scope failures and is cleared only by successful reconnect or explicit disconnect.

**Readiness classifier contract**: `consent_required`, `scope_missing`, and `access_revoked` are blocking readiness failures; transient/retryable provider or network failures are explicitly non-blocking for readiness.

**Slot-planning intent contract**: reconnect intent for attendee-inclusive slot planning must include attendee free/busy consent in requested scopes while still honoring least-privilege incremental consent.
