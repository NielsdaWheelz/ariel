# PR-01: Google OAuth Foundation + Read Flows

## Goal

Deliver a production-safe Google connector foundation plus read-only calendar/email journeys with deterministic typed auth/scope failures and clear recovery messaging.

## Builds On

Slice 3 PR-03 merged state (safe action runtime, surfaced lifecycle contracts, grounded response path).

## Acceptance

- users can complete Google connect, reconnect, status-check, callback-complete, and disconnect flows through dedicated connector endpoints, with clear connected/not-connected/reconnect-required status.
- OAuth flow is authorization-code + PKCE with CSRF-safe one-time state handles, strict callback validation, replay protection, and fail-closed behavior on invalid state/callback input.
- connector state is canonical and durable: account identity metadata, granted scopes, token expiry metadata, and encrypted token material persist safely without exposing secrets in user responses, events, or logs.
- connector lifecycle actions are auditable with explicit connector events and safe failure reasons, and disconnect immediately removes local token usability (with best-effort provider revoke).
- PR-01 requests only least-privilege read scopes required for calendar/email reads; missing broader consent is surfaced as reconnect guidance rather than pre-emptive over-scoping.
- given valid calendar read access, Ariel executes `cap.calendar.list` as allowlisted `read` without approval and returns user-visible schedule context with auditable lifecycle records.
- given slot-planning constraints, Ariel executes `cap.calendar.propose_slots` as allowlisted `read` without approval and returns concrete, user-visible slot options.
- attendee-aware slot proposals use attendee intersection when free/busy access is available; otherwise Ariel explicitly discloses the limitation, falls back to user-calendar-only planning, and provides a concrete recovery step.
- given Gmail read access, Ariel executes `cap.email.search` and `cap.email.read` as allowlisted `read` actions without approval, returns relevant in-chat results, and keeps lifecycle output redacted/auditable.
- calendar/email read requests that cannot execute due to connector state deterministically return exactly one typed recoverable class (`not_connected`, `consent_required`, `scope_missing`, `token_expired`, or `access_revoked`) plus a clear user recovery path and durable lifecycle reason.
- regression coverage demonstrates end-to-end connector lifecycle, read-flow success paths, attendee-aware fallback behavior, and each typed auth/scope failure class without approval-policy regressions.

## Non-Goals

- does not implement side-effecting Google actions (`cap.calendar.create_event`, `cap.email.draft`, `cap.email.send`).
- does not add Drive or Maps workflows.
- does not add proactive Google-triggered checks or notifications.
- does not add multi-user connector tenancy or autonomous background reconnect loops.

## Key Decisions (if any beyond L2)

**Connector audit boundary**: connector lifecycle events are first-class and durable, but connector-audit streams remain separate from turn/action timeline contracts to preserve clean event-domain boundaries.

**Typed failure contract shape**: calendar/email auth/scope failures are surfaced as structured typed outcomes (machine-readable class plus user recovery guidance), not string-parsed free text.

**`reconnect_required` semantics**: connector readiness is a derived user-action-required state that includes missing consent/scopes, revoked access, and non-recoverable token state, while excluding transient provider/network failures and self-recoverable refresh paths.
