# PR-02: Approval-Safe Writes (Calendar Create + Email Draft/Send)

## Goal

Complete Slice 4 write-path safety so calendar creation and email send remain approval-gated, while email drafting stays low-friction and non-sending by construction.

## Builds On

PR-01 merged.

## Acceptance

- write-path OAuth scope checks are capability-specific and least-privilege: missing write scopes return deterministic `consent_required` or `scope_missing` outcomes with clear reconnect guidance instead of over-broad up-front grants.
- reconnect flow for write remediation is capability-intent-driven: runtime computes and requests only scopes required for the attempted write behavior while preserving existing granted scopes.
- Ariel proposes `cap.calendar.create_event` as `write_reversible` with approval required; execution never starts before approval, and approved execution runs only once for the exact frozen payload hash.
- calendar-create outcomes are explicit and inspectable in surfaced lifecycle data as created/failed with auditable approval and execution events.
- Ariel executes `cap.email.draft` as allowlisted `write_reversible` without approval, persists canonical draft intent/content as draft-only state, and returns a user-visible drafted-not-sent result.
- draft execution never performs external delivery side effects.
- Ariel proposes `cap.email.send` as `external_send`; send execution always requires explicit approval and executes only once for the exact approved payload.
- draft and send remain separate action attempts with independently inspectable lifecycle/approval history so users can distinguish what was drafted from what was actually sent.
- send/create execution remains idempotent and race-safe under retries or concurrent approval requests; no duplicate external side effects are produced.
- typed connector/auth failures remain deterministic and recoverable across write paths (`not_connected`, `consent_required`, `scope_missing`, `token_expired`, `access_revoked`) with no silent fallback behavior.
- regression coverage proves write-path policy/approval semantics, incremental consent upgrade behavior, draft non-sending guarantees, and exact-once approved send/create execution.

## Non-Goals

- does not add auto-send, autonomous delivery, or approval bypass behavior.
- does not add advanced calendar management (series-wide edits, RSVP orchestration, resource booking, conflict optimization).
- does not add advanced Gmail automation beyond core search/read/draft/send.
- does not add Drive/Maps workflows or proactive Google-triggered notifications.

## Key Decisions (if any beyond L2)

**Capability-intent reconnect contract**: write-scope escalation is tied to the attempted write capability intent and remains least-privilege rather than requesting a blanket write-scope bundle.

**Canonical draft authority**: local persisted draft intent is the source of truth; provider draft state is a projection/reference and cannot blur draft-vs-send safety boundaries.

**Hard draft/send boundary**: drafting and sending are distinct lifecycle units with independent policy and approval semantics; drafting can never be treated as implicit send authorization.
