# PR-08: Fail-Closed Egress Preflight Enforcement

## Goal

Make egress controls truly execution-gating by validating outbound destinations before any external side effect can be dispatched.

## Builds On

PR-07 merged.

## Acceptance

- side-effecting capabilities use a two-phase execution boundary: egress intent is declared first, then outbound dispatch is permitted only after pre-dispatch policy validation.
- outbound side effects route through a centralized egress runtime boundary so policy enforcement is execution-gating rather than capability-implementation dependent.
- missing, malformed, undeclared, or non-allowlisted egress intent fails closed before external side effects, with deterministic auditable failure reasons.
- denied egress preflight paths produce zero external dispatch attempts and surface clear failure outcomes in user-visible lifecycle details.
- allowlisted/approved egress paths continue to execute successfully under existing approval and execution semantics.
- regression coverage proves denied preflight paths never dispatch while allowlisted paths still succeed under the same lifecycle and audit flow.
- surfaced message, timeline, and approval contracts remain redacted and allowlisted; internal egress/dispatch metadata does not leak.
- audit/event timelines remain reconstructable for preflight allow/deny decisions and final execution outcomes.

## Non-Goals

- does not provide OS-level sandboxing or host firewall guarantees.
- does not add new capability domains or domain-specific workflows.
- does not change approval token semantics, actor binding, or expiry behavior.
- does not redesign response-contract schemas or phone-surface UX structure.
- does not alter Slice 2 side-effect serialization/idempotency guarantees beyond adding pre-dispatch egress gating.

## Key Decisions (if any beyond L2)

**Preflight-declared egress is authoritative**: outbound dispatch must use declared destination intent that is policy-validated before dispatch; undeclared or ambiguous egress is blocked.

**Centralized dispatch boundary is mandatory**: side effects are executed through one egress-aware runtime choke point so enforcement remains durable as capability implementations evolve.

**Deny means zero dispatch**: an egress policy deny is a terminal execution failure with explicit user-visible and auditable reason, and no outbound attempt is made.
