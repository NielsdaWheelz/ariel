# PR-01: Capability Proposal + Deterministic Policy + Approval Lifecycle

## Goal

Introduce the Slice 2 action engine so Ariel can safely propose capability actions, authorize them deterministically, and execute only policy-allowed/approved actions.

## Builds On

Slice 1 PR-02 merged state.

## Acceptance

- model outputs proposing tool use are treated as untrusted and must pass capability existence/version + input-schema validation before execution.
- allowlisted low-impact `read` actions can execute without approval and return user-visible results with standard redaction.
- approval-required actions are persisted as pending approval and never execute before explicit approval.
- approval is single-use and bound to exact canonical action payload hash, actor identity, and expiry.
- approval execution runs the frozen proposed action payload (no re-plan/re-interpretation at approval time).
- denied or expired approvals do not execute and emit clear user-visible + auditable reasons.
- unknown/schema-invalid/policy-denied actions are blocked pre-execution with explicit audit events.
- action lifecycle observability is complete: proposed, policy decision, approval requested/decided, execution started/completed/failed, returned output/error.

## Non-Goals

- does not implement agency-domain action behavior (`cap.agency.*`) beyond framework hooks.
- does not implement calendar-domain action behavior (`cap.calendar.*`) beyond framework hooks.
- does not change Slice 1 conversation budget semantics.
- does not implement cross-session memory/session-rotation behavior.
- does not add batched, delegated, or adaptive auto-approval behavior.

## Key Decisions (if any beyond L2)

**Action-attempt state machine is separate from turn state**: proposal/approval/execution lifecycle is persisted independently from turn terminal status to support post-turn approval handling.

**Read-without-approval is constrained**: no-approval behavior applies only to explicit allowlisted low-impact read capabilities, not all read-like requests.
