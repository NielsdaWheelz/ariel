# PR-04: Surface Lifecycle Inspectability

## Goal

Ensure the phone-first surface exposes a complete, user-readable, redacted action lifecycle for each Slice 2 action attempt.

## Builds On

PR-03 merged.

## Acceptance

- users can inspect, for each action attempt in timeline/action details, a proposal payload summary, policy outcome and reason, approval outcome (including denied/expired reasons), and execution outcome with output/error.
- user-facing turn/timeline APIs expose a dedicated, allowlisted surface lifecycle view (not raw engine records) that the phone-first surface renders directly.
- the phone-first surface makes approval and execution outcomes visible directly in timeline/action details; users do not need internal logs to understand what happened.
- lifecycle details are coherent across inline reads and approval-gated executions, including terminal non-executed outcomes (`denied`, `expired`) and execution failures.
- surfaced outputs and reasons remain redacted per runtime redaction rules; secret-like values are not shown in timeline/action details.
- surfaced action lifecycle data excludes internal-only execution metadata while preserving user-relevant audit context.
- regression coverage demonstrates inspectability for key paths: inline read success, approval denied, approval expired, approval-approved execution success, and approval-approved execution failure.

## Non-Goals

- does not change Slice 2 authorization policy, taint semantics, or execution ordering/idempotency behavior.
- does not introduce visual redesign; this PR focuses on lifecycle completeness and clarity in the existing phone-first surface.
- does not add bulk/delegated approvals UX or workflow automation.
- does not add multi-user or tenancy changes.

## Key Decisions (if any beyond L2)

**Surface projection is API-enforced**: lifecycle inspectability is delivered through a server-side surfaced contract with explicit field allowlisting; client rendering is not a security boundary.
