# PR-06: Response Boundary Lock-In

## Goal

Prevent future metadata leakage by enforcing strict, surfaced-only response contracts on user-facing Slice 2 APIs.

## Builds On

PR-05 merged.

## Acceptance

- user-facing Slice 2 turn/timeline and approval responses are schema-enforced against explicit allowlisted contracts.
- non-allowlisted/internal lifecycle metadata cannot appear in user-facing responses, even if internal runtime/serialization shapes change.
- approval response payloads expose only surfaced approval lifecycle data needed for user flow continuity and audit clarity.
- surfaced lifecycle and approval payloads remain coherent and redacted across inline reads, pending approvals, denied approvals, expired approvals, approved execution success, and approved execution failure.
- compatibility/deprecation behavior is explicit and documented for removed legacy fields and legacy client expectations.
- contract tests fail when internal lifecycle metadata reappears or when surfaced response contracts drift.
- Slice 2 policy outcomes, approval semantics, capability contracts, and execution ordering/idempotency behavior remain unchanged.

## Non-Goals

- does not redesign authorization or taint-policy logic.
- does not change capability definitions, contract metadata, or executor guardrail behavior.
- does not redesign the phone surface beyond consuming the locked surfaced contracts.
- does not add new domain capabilities, tenancy changes, or autonomous background behavior.

## Key Decisions (if any beyond L2)

**Fail-closed response boundary**: user-facing Slice 2 responses are treated as strict contracts; internal fields are not recoverable through client-side filtering assumptions.

**Explicit legacy contract stance**: removed legacy fields are not silently reintroduced; compatibility/deprecation behavior is documented so client migration expectations are unambiguous.
