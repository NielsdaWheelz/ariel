# PR-01: Durable Single-Session Chat Skeleton

## Goal

Stand up a minimal Ariel backend + phone-friendly web chat that executes one model-backed turn and persists an auditable turn chain in Postgres.

## Builds On

Nothing - first implementation PR in Slice 0 (current repo baseline is docs-only).

## Acceptance

- user can send a message from the phone-friendly chat surface and receive a model-backed assistant response in that same turn.
- Ariel maintains exactly one active session, and each new message appends a new turn to that session.
- each turn persists an ordered event chain including turn start, model start/end, assistant response emission, and terminal completed/failed state.
- timeline history is rendered from the stored event chain (not ephemeral UI state) and shows turns in chronological order.
- model timeline entries include provider/model identity, duration, usage metadata when available, and explicit failure reason when present.
- model failures still produce a fully auditable terminal failed turn state with no dangling in-progress turn.
- persisted entities and user-visible API behavior follow constitution conventions for IDs, timestamps, standard error envelope, and auditability.

## Non-Goals

- does not implement private-tailnet ingress hardening or deployment lockdown (planned in PR-02).
- does not prove restart durability behavior in this PR (planned in PR-02).
- does not implement capability execution, approval workflows, async jobs, or durable cross-session memory.
- does not implement session rotation, multi-user tenancy, public internet hosting, or autonomous background actions.

## Key Decisions

**Event chain is the timeline source of truth**: user-visible timeline and audit views read from the same append-only persisted event stream to prevent divergence between chat and audit history.

**Single active-session contract is fixed for Slice 0**: one active session per user with append-only turn growth is the operating model; multi-session/rotation semantics are deferred.

**Every turn must terminate explicitly**: each turn must end as completed or failed, including model-error paths, so production state never contains orphaned turns.

**Model step is provider-agnostic but fully observable**: the turn engine records stable model-step metadata (provider/model/duration/usage/failure) to preserve portability and auditability as providers change.

**Durability is mandatory from day one**: Postgres-backed persistence for sessions, turns, and events is required; in-memory-only state is not acceptable for user-visible history.
